[Reading Notes] VerilogEval: Evaluating Large Language Models for Verilog Code Generation

DizzyHardSugar
2024-12-07

Motivation

LLMs have shown impressive capabilities to comprehend and generate human-like text. In the realm of Electronic Design Automation, LLMs have the potential to aid engineers in designing and verifying digital systems, providing insights into Verilog coding, optimizing circuits, and automating time-consuming tasks. Models like fine-tuned CodeGen and ChatGPT has initialed the evaluation of LLMs' potential in generating Verilog code and solution quality can be improved by simple and effective prompt engineering techniques. However, LLMs' pretraining phase using unsupervised training loss often lacks alignment with specific tasks, so supervised fine-tuning(SFT) is used to involve task-specific data to adapt to requirements, driving investigation of computationally demanding techniques such as reinforcement learning with human feedback. Acquiring labeled data is also cost-demanding, triggered a growing interest in alternative annotation-free alignment techniques. In the realm of Verilog coding, there remains a notable gap in the exploration of supervised fine-tuning for model enhancement. While there have been attempts to evaluate LLMs for Verilog, prior benchmarks like RTLLM focus on limited tasks or lack diversity in problem sets. They have problems like:

  1. lack of diversity(testing variance)

  2. ambiguous problem descriptions

  3. key factors: reliability and automation Hence the paper introduced a open-source benchmark, VerilogEval,

  4. A comprehensive evaluation dataset for diverse Verilog tasks.

  5. Automated methods to evaluate the functional correctness of generated Verilog code.

  6. Exploring the role of supervised fine-tuning (SFT) to improve LLMs’ Verilog generation performance.

Methods

1. Evaluation Framework: A Verilog Evaluation Set and Evaluate Metrics

They created a dataset with 156 diverse problems from HDLBits, covering tasks from simple combinational circuits to complex finite state machines, code debugging, and testbench construction. Used two methods to generate problem descriptions, Verilog-human and Verilog-machine. VerilogEval-human is manually curated descriptions from HDLBits problems, VerilogEval-machine is Machine-generated descriptions using GPT-3.5. The machine version are often more verbose, and also frequently delve into low-level details, tending to mirror the code’s implementation line by line, rather than focusing on the overarching functionality of the circuit. For Automated Evaluation Framework, the paper uses functional correctness testing to validate the generated Verilog code by simulating its behavior and comparing results with a golden solution. They also combines deterministic and randomized test patterns to ensure robustness. For Evaluation metric, the paper tried to use BLEU score, but recent works showed that it does not correlate well with functional correctness. This is because BLEU calculates similarity by comparing the n-grams (contiguous sequences of n words) between the generated text and the reference text, it only measures syntactic similarity, even if the generated code does not exactly match the reference code, it may still perform the same function. Instead they use pass@k score, evaluates the likelihood of generating at least one correct solution among kkk attempts.

2. Supervised Fine-tuning

They generated synthetic training data by pairing Verilog modules from GitHub with corresponding problem descriptions created by LLMs. Fine-tuned LLMs on this synthetic dataset to improve task alignment. Base-models:

  • codegen-nl: Natural language model. Trained on ThePile 825.18GB English text corpus.
  • codegen-multi: Code model. Initialized from codegen-nl and continue trained on BigQuery multi- lingual code dataset consisting of C, C++, Go, Java, JavaScript, and Python.
  • codegen-verilog: Verilog code model. Initialized from codegen-multi and continue trained on ∼300MB of Github Verilog and 400MB of textbook data.

Results:

  • More capable and larger models generally result in better Verilog coding capabilities.

  • Synthetically generated data yields notable enhancements in downstream model perfor- mance.

  • In the case of verilog models, VerilogEval- machine exhibited significant performance gains, whereas the VerilogEval-human approach displayed comparatively less improvement and, at times, even slight deteriorations.

  • Increasing SFT (and Verilog pretraining) data diversity and quality would further lead to increased performance. Overall,

  • Baseline Performance:

    • GPT-4 outperformed GPT-3.5 across both datasets.
    • Fine-tuned CodeGen models showed significant improvement over base models, particularly for VerilogEval-machine.

  • Impact of Fine-Tuning:

    • Supervised fine-tuning on the synthetic dataset improved performance but highlighted issues of overfitting and limited generalizability to human-crafted problems.
    • Fine-tuned models achieved comparable performance to GPT-3.5 but required less computational effort.

  • Model Scaling:

    • Larger models (e.g., CodeGen-16B) consistently outperformed smaller ones.
    • Base models pre-trained on Verilog data (CodeGen-verilog) showed better alignment with VerilogEval tasks.

Weakness

  • Limited Scope:
    • The benchmark evaluates standalone Verilog modules and does not address critical aspects like module instantiation or system-level design, which are essential in real-world hardware projects.
  • Synthetic Data Limitations:
    • Machine-generated descriptions lack the diversity and nuance of human-crafted examples, which may hinder generalizability.
  • Functional Testing Only:
    • The evaluation framework does not consider synthesizability or performance metrics (e.g., power, performance, and area optimizations) of the generated circuits.
  • Overfitting Risk:
    • Fine-tuning on synthetic data risks overfitting, as models may learn dataset-specific patterns rather than general principles.
